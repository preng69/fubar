DTF Protocol v0.1
=================

Status: draft
Transport: UDP
Purpose: peer-to-peer file discovery and byte-range file transfer


1. Overview
-----------

DTF is a UDP protocol for discovering files offered by nearby or known peers,
listing file metadata, and downloading specific byte ranges from those files.

The protocol is intentionally small:

- peers announce or query for files
- peers respond with files they can serve
- peers request byte ranges from files
- peers return file data in bounded UDP messages
- peers retry lost requests and ignore duplicate responses

DTF does not define authentication, authorization, encryption, reputation,
moderation, payment, or traffic policing. Implementations may add those outside
this specification, but they are not part of DTF v0.1.


2. Terminology
--------------

Peer:
    A process participating in the protocol.

Peer ID:
    A 16-byte random identifier generated by a peer and kept stable while the
    peer process is running.

File ID:
    A 32-byte identifier for a shared file. It MUST be the SHA-256 digest of
    the complete file bytes.

Chunk:
    A contiguous byte range from a file.

Offset:
    A zero-based byte position within a file.

Length:
    A number of bytes.

Request ID:
    A 64-bit random number chosen by the request sender. Responses copy the
    request ID so the sender can match responses to requests.

Session ID:
    A 64-bit value created by the peer that replies to HELLO. It lets both peers
    group later messages, but it is not a security mechanism.


3. UDP Transport
----------------

DTF uses UDP datagrams. The default port is 4747.

Peers MAY use unicast, multicast, or broadcast depending on the network
environment:

- unicast is used when a peer address is already known
- IPv4 broadcast MAY be used for LAN discovery
- multicast group 239.47.47.47 port 4747 MAY be used for LAN discovery

Each DTF message MUST fit inside one UDP datagram. Implementations SHOULD keep
datagrams at or below 1200 bytes to avoid IP fragmentation.

DTF messages are idempotent unless explicitly stated otherwise. A peer MAY
receive the same request multiple times and SHOULD send the same response again
when practical.


4. Byte Order and Primitive Types
---------------------------------

All fixed-width numeric fields use network byte order, also known as big-endian.

Primitive types:

- u8: unsigned 8-bit integer
- u16: unsigned 16-bit integer
- u32: unsigned 32-bit integer
- u64: unsigned 64-bit integer
- bytes[N]: exactly N bytes
- bytes: variable-length bytes
- string: u16 length followed by UTF-8 bytes

Boolean values are encoded as u8:

- 0 = false
- 1 = true


5. Packet Layout
----------------

Every DTF datagram starts with a fixed header:

    magic        bytes[4]   ASCII "DTF1"
    version      u8         1
    type         u8         message type
    flags        u16        message-specific flags, 0 if unused
    header_len   u16        header length in bytes, currently 44
    payload_len  u16        payload length in bytes
    request_id   u64        random request identifier
    session_id   u64        0 when not established
    sender_id    bytes[16]  sender peer ID

The header is followed by payload_len bytes of message-specific payload.

Receivers MUST ignore packets whose magic is not "DTF1", whose version is not
1, whose header_len is smaller than 44, or whose actual payload length does not
match payload_len.

Unknown message types MUST be ignored.


6. Message Types
----------------

Message type values:

    0x01 HELLO
    0x02 HELLO_ACK
    0x10 FIND_FILES
    0x11 FILES
    0x20 GET_RANGE
    0x21 RANGE_DATA
    0x22 RANGE_DONE
    0x30 CANCEL
    0x40 ERROR


7. Peer Handshake
-----------------

The handshake is optional for discovery and mandatory for range transfer.

7.1 HELLO

Purpose:
    Ask another peer to create or confirm a session.

Header:
    type = 0x01
    request_id = random
    session_id = 0 when no session exists

Payload:

    listen_port  u16
    name         string

listen_port is the UDP port on which the sender accepts DTF messages. name is a
human-readable peer name and MAY be empty.

7.2 HELLO_ACK

Purpose:
    Return the session ID selected by the responder.

Header:
    type = 0x02
    request_id = copied from HELLO
    session_id = responder-selected session ID

Payload:

    listen_port  u16
    name         string

The initiator SHOULD use the returned session_id in future GET_RANGE, CANCEL,
and related messages to that peer.


8. File Discovery
-----------------

Discovery is query-based. A peer sends FIND_FILES to one or more peers. Peers
that have matching files respond with FILES.

Discovery does not require an established session.

8.1 FIND_FILES

Purpose:
    Ask peers for files matching a query.

Header:
    type = 0x10
    request_id = random
    session_id = 0 or known session

Payload:

    query_kind      u8
    max_results     u16
    query           string

query_kind values:

    0 = list all available files
    1 = substring match against file name
    2 = exact file ID lookup; query is the SHA-256 file ID encoded as 64
        lowercase hexadecimal characters
    3 = metadata tag match

max_results is the maximum number of file records requested. If max_results is
0, the responder MAY choose its own limit.

query is interpreted according to query_kind. For list-all queries, query SHOULD
be empty.

8.2 FILES

Purpose:
    Return file records matching a FIND_FILES request.

Header:
    type = 0x11
    request_id = copied from FIND_FILES
    session_id = 0 or known session

Payload:

    total_matches   u32
    record_count    u16
    records         repeated FileRecord

FileRecord:

    file_id         bytes[32]
    file_size       u64
    chunk_size      u32
    name            string
    media_type      string
    tag_count       u16
    tags            repeated string

file_id is the SHA-256 digest of the complete file bytes. Peers MUST NOT use
any other identifier in this field. Two FileRecords with the same file_id refer
to the same complete byte sequence, regardless of name, media_type, or tags.

file_size is the total size of the file in bytes.

chunk_size is the responder's preferred range size for this file. Requesters MAY
request smaller or larger ranges, but responders MAY reject ranges that are too
large.

media_type SHOULD be an IANA media type such as "text/plain" or
"application/octet-stream". It MAY be empty.

total_matches is the number of matching files known to the responder, which MAY
be larger than record_count if the response was truncated.

If the FILES payload would exceed the datagram size limit, the responder SHOULD
include fewer records rather than fragmenting at the IP layer.


9. Range Download
-----------------

A file is downloaded by requesting byte ranges. Ranges are inclusive of the
starting offset and exclusive of the ending offset:

    [from_offset, to_offset)

The first byte of a file is offset 0. For a file with size N, the final valid
range ends at to_offset = N.

Requesters SHOULD avoid very large ranges. The recommended requested length is
16 KiB to 256 KiB. Responders split returned data into multiple RANGE_DATA
messages when needed.

9.1 GET_RANGE

Purpose:
    Request bytes from a file.

Header:
    type = 0x20
    request_id = random
    session_id = established session ID

Payload:

    file_id       bytes[32]
    from_offset   u64
    to_offset     u64
    max_datagram  u16

from_offset is inclusive. to_offset is exclusive.

max_datagram is the largest UDP payload the requester wants to receive for each
RANGE_DATA message, including the DTF header. A value of 0 means the responder
SHOULD use its default, which SHOULD be no more than 1200 bytes.

Rules:

- from_offset MUST be less than to_offset
- to_offset MUST be less than or equal to the file size
- the requested length is to_offset - from_offset
- the responder MAY return ERROR if the requested length is too large

9.2 RANGE_DATA

Purpose:
    Return part of a requested byte range.

Header:
    type = 0x21
    request_id = copied from GET_RANGE
    session_id = established session ID

Payload:

    file_id           bytes[32]
    requested_from    u64
    requested_to      u64
    data_offset       u64
    data_len          u16
    data_crc32        u32
    data              bytes

requested_from and requested_to copy the original requested range.

data_offset is the absolute file offset of the first byte in data.

data_len is the number of bytes in data and MUST match the remaining payload
length after the fixed RANGE_DATA fields.

data_crc32 is the IEEE CRC-32 checksum of data. It is used only to detect
accidental corruption.

Rules:

- data_offset MUST be greater than or equal to requested_from
- data_offset + data_len MUST be less than or equal to requested_to
- data_len MUST be greater than 0
- RANGE_DATA messages for one request MAY arrive out of order
- duplicate RANGE_DATA messages MUST be ignored after their bytes are accepted

9.3 RANGE_DONE

Purpose:
    Tell the requester that the responder has sent all data it intends to send
    for a GET_RANGE request.

Header:
    type = 0x22
    request_id = copied from GET_RANGE
    session_id = established session ID

Payload:

    file_id          bytes[32]
    requested_from   u64
    requested_to     u64
    sent_bytes       u64

RANGE_DONE is advisory. The requester MUST verify that it has received all bytes
in the requested range. If bytes are missing, it SHOULD issue new GET_RANGE
requests for the missing subranges.

Responders SHOULD send RANGE_DONE after all RANGE_DATA messages for a request
have been sent.

9.4 CANCEL

Purpose:
    Ask a responder to stop sending data for a previous GET_RANGE.

Header:
    type = 0x30
    request_id = random
    session_id = established session ID

Payload:

    cancelled_request_id  u64
    file_id               bytes[32]

Receivers MAY ignore CANCEL. Senders MUST tolerate receiving RANGE_DATA after
they send CANCEL.


10. Errors
----------

10.1 ERROR

Purpose:
    Report that a request could not be processed.

Header:
    type = 0x40
    request_id = copied from the failed request when available
    session_id = copied from the failed request when available

Payload:

    error_code    u16
    detail        string

error_code values:

    1  malformed message
    2  unsupported version
    3  unknown session
    4  file not found
    5  invalid range
    6  range too large
    7  temporarily unavailable
    8  unsupported query

detail is human-readable diagnostic text and MAY be empty.

Receivers MUST NOT depend on ERROR delivery. UDP may drop it.


11. Reliability Over UDP
------------------------

DTF does not make UDP reliable. Instead, requesters use retries and byte-range
repair.

Request IDs:

- Every request uses a random 64-bit request_id.
- Responses copy the request_id.
- A peer SHOULD remember recently completed request IDs for at least 60 seconds
  to identify duplicates.

Timeouts:

- HELLO and FIND_FILES SHOULD be retried after 500 ms if no response arrives.
- GET_RANGE SHOULD be considered incomplete if no new RANGE_DATA arrives for
  1000 ms.
- Implementations MAY adapt these values based on latency.

Retry limits:

- HELLO SHOULD be attempted up to 3 times per peer.
- FIND_FILES SHOULD be attempted up to 2 times per peer or discovery address.
- GET_RANGE SHOULD be retried by requesting only missing byte subranges.

Out-of-order delivery:

- Requesters MUST accept RANGE_DATA in any order.
- Requesters SHOULD track received intervals for each request.
- Requesters SHOULD write data to the target file only after data_crc32 matches.

Duplicate delivery:

- Duplicate RANGE_DATA for bytes already received SHOULD be ignored.
- Duplicate FIND_FILES and GET_RANGE requests MAY be answered again.

Partial completion:

- A requester MUST treat a range as complete only when every byte in
  [from_offset, to_offset) has been received and passed its CRC-32 check.
- RANGE_DONE alone is not proof of completion.


12. File Integrity
------------------

FileRecord.file_id MUST be the SHA-256 digest of the complete file bytes. This
is required for integrity checking and for downloading different byte ranges of
the same file from multiple peers.

After all ranges for a file have been downloaded, the requester MUST compute
SHA-256 over the assembled file and compare it to file_id.

If the digest does not match, the downloaded file MUST be treated as invalid.
The requester MAY retry ranges from the same peer or from other peers.

RANGE_DATA.data_crc32 detects accidental corruption in individual datagrams
before the full-file digest is available.


13. Recommended Download Flow
-----------------------------

1. Generate a random 16-byte peer ID.
2. Send FIND_FILES to known peers, broadcast, or multicast.
3. Receive FILES responses and choose one or more file records.
4. Send HELLO to peers that have the selected file.
5. Receive HELLO_ACK and store session IDs per peer.
6. Split the target file into byte ranges.
7. Send GET_RANGE requests for needed ranges.
8. Accept RANGE_DATA in any order, checking data_crc32.
9. Use RANGE_DONE and local interval tracking to find missing bytes.
10. Re-request missing subranges until the file is complete or peers stop
    responding.
11. Compute SHA-256 of the assembled file and compare it to file_id.


14. Serving Flow
----------------

1. Generate a random 16-byte peer ID.
2. Maintain a local index of shared files.
3. Listen for DTF UDP datagrams.
4. Respond to FIND_FILES with matching FILES records.
5. Respond to HELLO with HELLO_ACK and a session ID.
6. Respond to GET_RANGE by reading the requested file bytes.
7. Split the requested range into RANGE_DATA datagrams that fit max_datagram.
8. Send RANGE_DONE after the selected data has been sent.
9. Respond with ERROR for invalid requests when useful.


15. Example Messages
--------------------

Example: discover all files

    Header:
        type = FIND_FILES
        request_id = 812312333991
        session_id = 0

    Payload:
        query_kind = 0
        max_results = 25
        query = ""

Example: request bytes 1048576 through 1114111

    Header:
        type = GET_RANGE
        request_id = 998812300120
        session_id = 441200991

    Payload:
        file_id = 32-byte SHA-256 digest
        from_offset = 1048576
        to_offset = 1114112
        max_datagram = 1200

The requested range contains 65536 bytes because to_offset is exclusive.


16. Versioning
--------------

The version field is 1 for this specification.

Future versions MUST use a different version number. Peers that receive an
unsupported version MAY send ERROR with error_code 2 if the message is readable
enough to determine the sender address and request ID.

Flags are reserved unless defined by a later specification. Senders MUST set
unknown flags to 0. Receivers MUST ignore unknown flags.
